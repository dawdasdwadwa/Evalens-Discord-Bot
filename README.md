# Evalens Discord Bot

Discord-бот сервера Evalens. Плоская структура — все `.py`-файлы лежат в
корне репозитория (без `cogs/`, `config/`, `utils/`), деплой на Railway.

## Функциональность

1. **Приветствие новых участников** (`welcome.py` + `image_generator.py`)
   При заходе участника автоматически выдаётся роль и в канал приветствия
   отправляется сгенерированная через Pillow PNG-карточка с пингом
   пользователя.

2. **Верификация по кнопке** (`verification.py`)
   Панель верификации публикуется автоматически при каждом старте/редеплое
   бота (старые сообщения бота в канале чистятся перед публикацией новой
   панели). Успешные верификации логируются отдельно.

3. **Логи приглашений** (`invite_logs.py`)
   При заходе участника определяется, по какому инвайту он зашёл (сравнение
   счётчиков использований до/после), и логируется: кто зашёл, чей инвайт,
   код, число использований, когда и кем создана ссылка. Для vanity-ссылок
   и виджета Discord API не отдаёт эту информацию — бот честно пишет
   "не удалось определить". Требует право бота **Manage Server**.

4. **Модерация** (`moderation.py`)
   Слэш-команды `/ban`, `/unban`, `/mute`, `/unmute`, `/server`, `/profile`,
   `/очистить`. Доступны только участникам с ролями из `STAFF_ROLE_IDS`
   (`/profile` — из `PROFILE_ROLE_IDS`).
   - `/ban` — бан с причиной и произвольным сроком (`10m`, `2h`, `3d`, `1w`
     или `навсегда`). Временные баны хранятся в `temp_bans.json` и снимаются
     автоматически фоновой задачей.
   - `/mute` — мьют через нативный Discord timeout (макс. 28 дней — это
     лимит самого Discord API, не бота).
   - `/unban`, `/unmute` — снятие бана/мьюта с указанием причины.
   - `/server` — статистика сервера (участники, каналы, роли, бусты).
   - `/profile` — профиль участника (аватар, дата создания аккаунта, дата
     захода на сервер, роли).
   - `/очистить` — удаляет указанное количество сообщений в канале.
   - **Автомодерация**: ссылки-приглашения на чужой Discord, спам
     одинаковыми сообщениями, флуд и спам пингами — участнику автоматически
     выдаётся мьют на 10 минут с уведомлением в ЛС, действие логируется и
     видно всем в чате.
   - Баны/разбаны логируются в `BAN_LOG_CHANNEL_ID`, мьюты/размьюты — в
     `MUTE_LOG_CHANNEL_ID`.

5. **Система тикетов** (`tickets.py`)
   В канале `TICKETS_CHANNEL_ID` публикуется панель «Тикеты» с двумя
   кнопками: **Жалоба на игрока** и **Жалоба на персонал**. При нажатии
   открывается модальное окно (используется нативный Discord-компонент
   загрузки файлов прямо в модалке):
   - «Жалоба на игрока» — поля «Ваш ник», «Ник нарушителя», «Причина»,
     «Доказательства» (до 3 файлов).
   - «Жалоба на персонал» — поля «Ваш ник», «Ник нарушителя», «Должность
     сотрудника», «Причина», «Доказательства» (до 3 файлов). При создании
     такого тикета пингуются только роли из `STAFF_REPORT_PING_ROLE_IDS`.

   После отправки формы бот создаёт приватный текстовый канал-тикет,
   видимый только автору обращения и ролям из `STAFF_ROLE_IDS`, с эмбедом
   всех ответов и прикреплёнными файлами, в соответствующей "открытой"
   категории (`PLAYER_REPORT_OPEN_CATEGORY_ID` / `STAFF_REPORT_OPEN_CATEGORY_ID`).

   Внутри тикета четыре кнопки (все — только для стаффа):
   - **Рассмотреть** — отправляет в канал эмбед-уведомление, что заявку
     рассматривает администратор, с пингом того, кто нажал кнопку.
   - **Закрыть тикет** — не удаляет канал, а переносит его в
     соответствующую "закрытую" категорию (`PLAYER_REPORT_CLOSED_CATEGORY_ID` /
     `STAFF_REPORT_CLOSED_CATEGORY_ID`) и закрывает доступ к каналу автору
     тикета, а также ролям из `TICKET_CLOSED_HIDDEN_ROLE_IDS` — они больше
     не видят закрытый тикет. После закрытия автор тикета может в любой
     момент создать новый тикет того же типа — проверка "уже есть открытый
     тикет" учитывает только каналы в "открытой" категории.
   - **Добавить пользователя** — открывает меню выбора участника; выбранный
     пользователь получает доступ на чтение канала (видит переписку, но не
     может писать).
   - **Удалить тикет** — полностью и безвозвратно удаляет канал тикета
     (в отличие от «Закрыть тикет», который архивирует канал).

   Открытие, закрытие, удаление тикетов и добавление пользователей подробно
   логируются в `TICKET_LOG_CHANNEL_ID`: кто создал тикет, тип и содержимое
   анкеты (ник, причина и т.д.), кто и когда закрыл/удалил тикет или кого
   добавили. Лимит тикетов на участника — `TICKET_DAILY_LIMIT` (по умолчанию
   10) в сутки.
   Требует право бота **Manage Channels**.

6. **Логи ролей** (`role_logs.py`)
   В канал `ROLE_LOG_CHANNEL_ID` подробно логируется всё, что происходит с
   ролями на сервере:
   - кому какая роль выдана и кем (участник, роль, кто выдал);
   - у кого какая роль снята и кем;
   - создание новой роли на сервере (кем создана);
   - удаление роли с сервера (кем удалена).

   Чтобы бот мог определить, **кто именно** выдал/снял/создал/удалил роль,
   ему требуется право **View Audit Log** (Просмотр журнала аудита) — без
   него сам факт изменения всё равно залогируется, но поле "Кто изменил"
   будет содержать "не удалось определить".

## Структура проекта

```
.
├── bot.py               # точка входа, грузит коги
├── settings.py          # все настройки через переменные окружения
├── welcome.py           # приветствие + автовыдача роли при входе
├── verification.py      # автопубликация панели, кнопка, логи верификации
├── invite_logs.py       # определение инвайта и логирование
├── moderation.py        # /ban /unban /mute /unmute /server /profile /очистить + автомодерация
├── tickets.py            # система тикетов (панель + модалка + приватные каналы)
├── role_logs.py          # подробные логи выдачи/снятия/создания/удаления ролей
├── image_generator.py   # генерация PNG-карточки (Pillow)
├── Poppins-*.ttf         # шрифты для карточки
├── requirements.txt
├── Procfile
├── .env.example
└── .gitignore
```

## Установка

```bash
pip install -r requirements.txt
cp .env.example .env
```

Заполни `.env` (или, для продакшена, задай эти же переменные в
Railway → Variables — **не в самом репозитории**):

- `DISCORD_TOKEN` — токен бота (Discord Developer Portal → Bot → Token)
- `VERIFIED_ROLE_ID` — роль, выдаваемая после верификации (обязательно)
- `UNVERIFIED_ROLE_ID` — роль "новичок", снимается после верификации (опционально)

Остальные переменные (ID каналов и ролей) уже захардкожены как значения
по умолчанию в `settings.py` и переопределять их не обязательно —
но при желании их тоже можно вынести в `.env` (см. `.env.example`).

## Права и intents (Discord Developer Portal → Bot)

Privileged Gateway Intents — включить оба:
- **SERVER MEMBERS INTENT** — обязательно для приветствия и автовыдачи роли
- **MESSAGE CONTENT INTENT** — обязательно для автомодерации ссылок-приглашений в `moderation.py`

Права роли бота на сервере (роль бота должна быть **выше** ролей, которыми
она управляет — верификационной, замьюченных/забаненных участников):
- `Manage Roles`
- `Manage Server` (для `invite_logs.py`)
- `Manage Channels` (для `tickets.py` — создание приватных каналов-тикетов)
- `Ban Members`, `Moderate Members` (для `moderation.py`)
- `View Audit Log` (для `role_logs.py` — определить, кто выдал/снял/создал/удалил роль)
- `Send Messages`, `Embed Links`, `Attach Files`, `Manage Messages`
- `Use Application Commands`

## Запуск

```bash
python bot.py
```

## Деплой на Railway

1. Подключить репозиторий к Railway
2. Добавить переменные окружения в настройках проекта
3. Railway автоматически определит `requirements.txt` и запустит `Procfile` (`worker: python bot.py`)

Если используются временные баны (`/ban` со сроком), для надёжности
рекомендуется подключить Railway Volume — иначе `temp_bans.json`
сбрасывается при каждом редеплое и текущие временные баны не снимутся
автоматически (придётся `/unban` вручную).
